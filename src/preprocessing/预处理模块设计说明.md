# 预处理模块设计说明

## 1. 模块概述

预处理模块负责接收摄像头实时视频流或本地视频文件，对输入视频进行逐帧解析与基础处理，并为后续特征提取模块和界面展示模块提供统一、稳定、可直接消费的数据。

本模块对应系统概要设计中的“预处理模块”，核心职责如下：

- 完成视频采集与视频解码
- 对视频帧进行标准化处理
- 过滤空帧、损坏帧等异常输入
- 检测当前帧中的全部人脸
- 在连续帧之间维持稳定的人脸编号
- 对检测到的人脸进行活体有效性判断
- 从多张人脸中确认主人脸
- 分别输出界面展示数据与特征提取数据

当前实现采用“新增文件、不改动原有代码”的方式落地，便于独立开发、后续集成和责任划分。

## 2. 设计目标

本模块的设计目标如下：

- 满足概要设计文档规定的输入输出格式
- 提供清晰的内部层次结构，便于单人负责与多人协作
- 保证 `face_id` 在连续帧中的相对稳定
- 在模型缺失或部分依赖不可用时提供降级方案
- 在连续处理失败时具备基础恢复能力
- 保持与特征提取模块低耦合，仅通过约定数据结构通信

## 3. 输入与输出设计

### 3.1 输入

模块支持两类输入源：

- 摄像头实时视频流
- 本地视频文件

输入数据通过 `cv2.VideoCapture` 解码后，逐帧进入预处理管线。

### 3.2 输出

本模块按概要设计要求提供两条独立输出通道。

#### 输出到界面展示模块

```python
{
    "frame": frame,
    "faces": [
        {"face_id": 1, "bbox": [320, 200, 120, 150]},
        {"face_id": 3, "bbox": [800, 180, 110, 140]}
    ],
    "timestamp": 1734567890.123
}
```

字段含义如下：

- `frame`：当前原始视频帧，BGR 格式
- `faces`：当前帧中的全部人脸框与编号
- `timestamp`：当前帧时间戳

#### 输出到特征提取模块

```python
{
    "timestamp": 1734567890.123,
    "faces": [
        {"face_id": 1, "face_roi": face_img1},
        {"face_id": 2, "face_roi": face_img2}
    ],
    "owner_face_id": 1,
    "frame": frame
}
```

字段含义如下：

- `timestamp`：当前帧时间戳
- `faces`：包含所有有效人脸的裁剪结果
- `owner_face_id`：主人脸编号，无主人脸时为 `-1`
- `frame`：当前原始视频帧，BGR 格式

当当前帧无人脸或无人脸通过活体检测时：

- `faces = []`
- `owner_face_id = -1`
- `frame` 保留正常输出

## 4. 总体结构设计

当前预处理模块被拆分为以下几个子文件：

- `contracts.py`：定义模块内部统一使用的数据结构
- `video_source.py`：封装摄像头和视频文件输入
- `face_tracker.py`：实现跨帧人脸编号跟踪
- `liveness.py`：实现轻量活体检测
- `pipeline.py`：组织预处理主流程
- `service.py`：对外提供命令式服务接口
- `README.md`：模块使用说明

该设计遵循“输入源层 - 算法处理层 - 服务控制层”的分层思路。

## 5. 详细设计

### 5.1 数据结构设计

为保证模块边界清晰，系统对关键中间数据做了结构化封装。

#### `FrameContext`

用于表示从视频源读取出的单帧上下文信息，包含：

- `frame`：视频帧
- `timestamp`：时间戳
- `source_name`：输入源名称
- `frame_index`：帧编号

#### `DetectionResult`

用于表示单张人脸检测结果，包含：

- `bbox`：人脸框坐标
- `confidence`：检测置信度
- `face_roi`：裁剪并缩放后的人脸图像

#### `TrackedFace`

用于表示完成跟踪和活体判断后的人脸对象，包含：

- `face_id`：跨帧稳定编号
- `bbox`：人脸框
- `confidence`：检测置信度
- `face_roi`：人脸区域图像
- `is_live`：是否为活体
- `tracking_score`：跟踪稳定度得分

#### `UIFramePacket` 与 `FeatureFramePacket`

分别对应界面模块和特征提取模块的最终输出格式，避免调用方手工拼接字段。

### 5.2 视频输入设计

`video_source.py` 中的 `VideoSource` 负责统一管理两类视频源。

主要接口如下：

- `open_camera(device_id)`：打开指定摄像头
- `open_file(file_path)`：打开本地视频文件
- `read()`：读取下一帧
- `close()`：关闭视频源

设计理由如下：

- 将输入源处理与算法逻辑分离，便于后续扩展网络流或其他输入形式
- 所有输入源都被统一封装成 `FrameContext`，便于主流程复用

### 5.3 帧标准化设计

在 `PreprocessingPipeline` 中，读取到的原始帧首先进入标准化阶段，主要处理包括：

- 校验是否为空帧或非法对象
- 将图像统一缩放为 `1280 x 720`
- 若输入为灰度图，则转换为 BGR 三通道格式

这样做的目的有三点：

- 保持后续检测输入尺寸稳定
- 简化界面层与特征层的兼容处理
- 降低不同摄像头或视频格式带来的分支逻辑

### 5.4 异常帧过滤设计

当出现以下情况时，当前帧会被判定为无效帧：

- `frame is None`
- 输入不是 `numpy.ndarray`
- 帧尺寸为空或内容损坏

对无效帧，系统不会继续做人脸检测，而是：

- 记录 `invalid_frames`
- 增加连续失败计数
- 记录最近一次错误信息

### 5.5 人脸检测设计

人脸检测由 `pipeline.py` 中的 `FaceDetector` 负责，采用双层方案：

1. 优先使用 `MTCNN`
2. 若 `MTCNN` 不可用或返回空结果，则回退到 OpenCV Haar Cascade

这样设计的原因是：

- `MTCNN` 对人脸检测精度更高，更贴近系统概要设计中的预训练模型路线
- Haar Cascade 无需额外深度学习推理环境，适合作为降级方案
- 双层方案提高了模块可运行性和演示稳定性

检测后，每张人脸都会被裁剪成固定大小的 `face_roi`，当前实现统一为 `224 x 224`。

### 5.6 人脸跟踪与编号设计

人脸跟踪由 `face_tracker.py` 中的 `SimpleFaceTracker` 完成。

当前实现采用基于中心点距离的轻量跟踪策略：

- 计算当前检测框中心点
- 与历史轨迹的中心点做距离匹配
- 若距离小于阈值，则认为是同一张人脸
- 若没有匹配轨迹，则分配新的 `face_id`
- 连续多帧未匹配的轨迹会被清除

该方案虽然不如深度排序跟踪算法复杂，但适合本项目当前阶段，具有以下优点：

- 实现简单，便于课程项目说明与维护
- 无需额外模型依赖
- 对单人网课场景和少量人脸场景基本够用

### 5.7 活体检测设计

活体检测由 `liveness.py` 中的 `HeuristicLivenessDetector` 实现。

当前未引入单独活体神经网络模型，而是采用启发式方案：

- 先检查人脸区域尺寸是否过小
- 将人脸转为灰度图
- 计算 Laplacian 方差，衡量图像纹理与清晰度
- 结合亮度标准差生成活体得分

判断逻辑如下：

- 若人脸尺寸过小，则判定为低可信目标
- 若纹理变化不足，则判定为非活体或低质量目标
- 满足阈值的人脸才会进入特征提取输出通道

该设计的优点是：

- 便于本地离线运行
- 不增加新的模型依赖
- 能在课设实现中体现“活体检测”功能点

不足之处是：

- 对打印照片、屏幕翻拍等复杂攻击方式识别能力有限
- 更适合作为轻量占位方案，而非高安全场景最终方案

### 5.8 主人脸确认设计

主人脸确认是本模块的关键职责之一。

当前实现将所有通过活体检测的人脸作为候选，并综合以下因素计算得分：

- 人脸框面积占比
- 人脸中心与画面中心的距离
- 跟踪稳定度
- 与上一帧主人脸是否连续

最终得分最高的人脸被选为当前帧的 `owner_face_id`。

这样设计的原因是：

- 网课场景通常以正中间、最大的人脸作为主要学生目标
- 加入连续性奖励后，可以减少主人脸频繁跳变
- 不依赖身份识别模型，适合当前项目阶段

当没有任何有效活体人脸时，系统输出：

- `owner_face_id = -1`

### 5.9 服务接口设计

`service.py` 中的 `PreprocessingService` 是模块的对外服务入口。

其主要职责包括：

- 接收外部控制命令
- 打开或关闭视频源
- 在后台线程中持续处理帧
- 将处理结果分发给界面模块和特征提取模块

支持的命令如下：

- `toggle_capture`
- `load_video`
- `query_cameras`

命令式接口的好处在于：

- 与现有界面模块中的回调式设计风格一致
- 后续接入 `interface_manager` 时改动最小
- 便于单独测试和脚本调用

### 5.10 异常恢复设计

根据概要设计文档，本模块需要具备连续失败后的恢复能力。

当前实现中，系统维护以下统计信息：

- 总读取帧数
- 成功处理帧数
- 无效帧数
- 检测失败次数
- 连续失败次数
- 恢复次数
- 最近一次错误信息

当连续失败达到阈值后，系统执行恢复逻辑：

- 重置跟踪器
- 清空主人脸状态
- 记录恢复事件

该方案实现了概要设计中“连续失败超过阈值时重新初始化”的要求。

## 6. 处理流程说明

当前模块的整体处理流程如下：

1. 打开摄像头或本地视频文件
2. 使用 `cv2.VideoCapture` 逐帧读取视频
3. 对每帧做标准化与异常检查
4. 执行人脸检测
5. 对检测到的人脸进行跨帧跟踪并分配稳定 `face_id`
6. 对每张人脸执行活体检测
7. 裁剪有效人脸区域，生成 `face_roi`
8. 从活体人脸中选出主人脸
9. 组装界面输出包
10. 组装特征提取输出包
11. 通过回调发送到后续模块

## 7. 与其他模块的接口关系

### 7.1 与界面模块的关系

预处理模块向界面模块输出：

- 原始视频帧
- 所有人脸框
- 对应 `face_id`
- 时间戳

界面模块可据此完成：

- 实时视频展示
- 人脸框绘制
- 编号显示

### 7.2 与特征提取模块的关系

预处理模块向特征提取模块输出：

- 当前帧时间戳
- 所有有效人脸 `face_roi`
- 主人脸编号 `owner_face_id`
- 原始帧

特征提取模块仅需根据 `owner_face_id` 找到目标人脸，即可继续进行面部特征与姿态特征提取。

## 8. 方案优点与局限

### 8.1 优点

- 结构清晰，符合课设模块划分要求
- 与概要设计数据接口保持一致
- 不改动原仓库已有文件，集成风险低
- 有基础降级与恢复机制，便于展示稳定性
- 轻量实现便于答辩时解释

### 8.2 局限

- 跟踪算法为轻量启发式方案，复杂遮挡场景下稳定性有限
- 活体检测为启发式实现，安全性不如专用模型
- 主人脸确认未结合身份识别，仅基于几何与时序信息
- 当前只完成预处理模块本体，尚未与后续模块正式接线

## 9. 后续可优化方向

后续若继续完善，可从以下方向迭代：

- 替换为更稳定的多目标跟踪算法
- 引入专用活体检测模型
- 增加基于身份特征的主人脸确认
- 支持配置文件化参数管理
- 补充独立测试样例与演示脚本
- 增加日志持久化与异常告警接口

## 10. 结论

本预处理模块设计已经覆盖系统概要设计文档中要求的主要职责，并通过新增代码文件的形式完成了独立实现。模块在结构上清晰分层，在功能上具备输入处理、人脸检测与跟踪、活体判断、主人脸确认、双通道输出和异常恢复能力，能够作为后续特征提取模块的稳定前置模块使用。
